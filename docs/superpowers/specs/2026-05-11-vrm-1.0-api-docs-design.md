# VRMMetalKit 1.0 — API Documentation Design

**Status:** Approved
**Date:** 2026-05-11
**Author:** Paul Flynn (paul@arkavo.com), with Claude
**Scope:** Documentation for the 1.0 release. No source-level API changes.

## Goal

Ship a complete, navigable API reference for VRMMetalKit alongside the 1.0 tag, so library consumers get Xcode Quick Help, an in-editor symbol graph, and a hosted reference site without having to read source.

## Non-goals

- Renaming, restructuring, or removing public API.
- Migrating the existing `docs/` engineering archive (ADRs, SHADERS.md, PHYSICS_REVIEW, performance reports, internal specs).
- Tutorials beyond a Getting Started article. Long-form tutorials are post-1.0.
- Localized docs.

## Approach: hybrid

Add a Swift-DocC catalog that owns the **public API reference and user-facing articles**. Leave `docs/` untouched as the engineering archive. The two surfaces serve different audiences:

| Surface           | Audience                  | Format         |
| ----------------- | ------------------------- | -------------- |
| DocC catalog      | Library consumers         | DocC bundle    |
| `docs/` directory | Contributors, maintainers | Markdown notes |

Rejected alternatives:

- **Full migration into DocC.** High churn, risks breaking inbound links, and DocC is a poor home for ADRs and internal performance reports.
- **Markdown-only (expand README + docs/).** No Xcode Quick Help, no symbol cross-linking, no hosted reference site. Out of step with Apple-platform expectations and harder for new consumers to discover.

## Catalog layout

```
Sources/VRMMetalKit/VRMMetalKit.docc/
├── VRMMetalKit.md                  # Module landing page (@Metadata + Topics)
├── Resources/
│   └── images/                     # diagrams, sample renders
└── Articles/
    ├── GettingStarted.md
    ├── LoadingVRMModels.md         # VRMModel, VRMLoadingOptions, security limits
    ├── RenderingAvatars.md         # VRMRenderer, RendererConfig, MSAA, MToon
    ├── AnimationAndRetargeting.md  # AnimationPlayer, VRMA loader, retargeting model
    ├── ARKitIntegration.md         # ARKitFaceDriver, ARKitBodyDriver, smoothing
    ├── SpringBonePhysics.md        # XPBD, colliders, tuning guidance
    ├── StrictMode.md               # Validation levels, when to enable
    └── MigratingFromVRM0.md        # 0.x → 1.0 behavioral differences users will hit
```

### Landing page (`VRMMetalKit.md`)

- One-paragraph module overview.
- `## Overview` section with the "what / why / when" framing.
- `## Topics` groups organizing the public surface:
  - **Essentials** — `VRMMetalKit`, `VRMModel`
  - **Loading** — `VRMLoadingOptions`, loader entry points, error types
  - **Rendering** — `VRMRenderer`, `RendererConfig`, `VRMPipelineCache`
  - **Materials** — MToon types, material report
  - **Animation** — `AnimationPlayer`, `VRMAnimationLoader`, retargeting helpers, IK, layers
  - **Expressions and Gaze** — expression controller, mixer, look-at controller
  - **Physics** — SpringBone systems, colliders, configuration
  - **ARKit** — face / body drivers, smoothing filters, coordinate converter
  - **Performance** — `PerformanceMetrics`, character priority system, frustum culler
  - **Validation** — `StrictMode`
  - **Errors** — `VRMRendererError` and any other public error types

### Articles

Each article: 200–600 words, one runnable Swift snippet, links into the symbol graph with double-backtick references. Articles answer "how do I do X with the public API" — they are not internals documentation.

## Symbol documentation pass

Every `public` type, init, method, property, and case in `Sources/VRMMetalKit/` gets at minimum a one-line `///` summary. The `.metal` shader sources (already excluded from the SPM build in `Package.swift`) and the `Resources/` directory are out of scope. Swift files inside `Sources/VRMMetalKit/Shaders/` (e.g., `MToonShader.swift`) that expose public types are **in scope**.

Priority order for the deeper treatment (full Discussion + See Also blocks):

1. `VRMMetalKit` (module facade)
2. `VRMModel` and loader entry points
3. `VRMRenderer`, `RendererConfig`
4. `AnimationPlayer`, `VRMAnimationLoader`
5. `ARKitFaceDriver`, `ARKitBodyDriver`
6. `VRMLoadingOptions` (security-sensitive surface)
7. `StrictMode`
8. `VRMRendererError` and other public errors

Comment shape:

```swift
/// One-line summary that renders in Quick Help.
///
/// Optional discussion paragraph. Explain the why, not the what.
/// Cross-link related symbols with double backticks: ``VRMRenderer``.
///
/// - Parameters:
///   - device: The Metal device used for resource creation.
///   - config: Renderer configuration. See ``RendererConfig`` for defaults.
/// - Throws: ``VRMRendererError/pipelineCompilationFailed`` if shader linking fails.
/// - Returns: A configured renderer ready to draw frames.
```

Doc style rules:
- Lead with verbs ("Loads…", "Renders…", "Configures…").
- No "the" at the start of summaries unless grammatically required.
- Document `throws` cases by symbol, not freeform prose.
- Use `## See Also` sparingly — only when the user genuinely needs the other symbol to use this one.
- Match the codebase rule from CLAUDE.md: no contextual chatter about the current task, callers, or history.

## Toolchain wiring

Add `apple/swift-docc-plugin` to `Package.swift` so DocC tooling is available without affecting library consumers.

```swift
// Package.swift
dependencies: [
    .package(url: "https://github.com/apple/swift-docc-plugin", from: "1.4.0")
],
```

`Makefile` additions:

```make
docs:           ## Build and preview docs locally
	swift package --disable-sandbox preview-documentation --target VRMMetalKit

docs-static:    ## Build a static site under .build/docs
	swift package --disable-sandbox \
		--allow-writing-to-directory .build/docs \
		generate-documentation --target VRMMetalKit \
		--disable-indexing \
		--transform-for-static-hosting \
		--hosting-base-path VRMMetalKit \
		--output-path .build/docs
```

GitHub Actions workflow at `.github/workflows/docs.yml`:

- Triggers: `workflow_dispatch` (manual) and `push` of tags matching `v*`.
- Runs `make docs-static` on macOS runner.
- Publishes `.build/docs` to the `gh-pages` branch via the standard `peaceiris/actions-gh-pages` (or equivalent) action.
- `hosting-base-path` is `VRMMetalKit` so the site lives at `https://arkavo-org.github.io/VRMMetalKit/`.

Manual trigger is included so the first publish does not have to wait for the 1.0 tag.

## README slim-down

The README stays as the GitHub-front-page funnel. Changes:

- Keep: project tagline, install snippet, badge row, license, links to LICENSE-MODELS and SECURITY.
- Keep: the `VRMRender` CLI section — it is a separately useful entry point.
- Move into DocC `GettingStarted.md`: the multi-step "Basic Setup" / "Quick Start" code blocks currently in README.
- Replace the moved content with a 3-line "Get started" pointer to the DocC site.
- Net effect: README becomes ~250 lines instead of 728, all factual material preserved in DocC.

## What does not change

- `docs/` remains the engineering archive. Add a top-of-file note to `docs/README.md` (create if missing): "User-facing API docs live in DocC. Files here are engineering reference: ADRs, performance reports, internal specs, and contributor guides."
- No symbol renames, no signature changes, no removals.
- No new test targets or test obligations beyond the existing CI.

## 1.0 completeness gate

The 1.0 tag ships when:

1. The DocC catalog exists at `Sources/VRMMetalKit/VRMMetalKit.docc/` and builds without errors.
2. Every `public` declaration in `Sources/VRMMetalKit/` has at minimum a one-line `///` summary.
3. The landing page `## Topics` groups cover all public symbols (no orphans).
4. All eight articles exist as complete first drafts meeting the "Articles" section criteria (200–600 words, one runnable Swift snippet, links into the symbol graph with double-backtick references). Stubs do not count.
5. `make docs-static` produces a static site that loads in a browser without console errors.
6. The GitHub Pages workflow has run successfully at least once via `workflow_dispatch`.

Explicitly **not** required for 1.0:

- `-warnings-as-errors` for missing doc comments. Defer to a follow-up issue so PRs mid-rollout are not blocked.
- Deep Discussion blocks on every public symbol — the priority list above is the cut.
- Article screenshots and diagrams beyond what exists in `Sources/VRMMetalKit/Resources/` or the repo's existing PNGs. Add post-1.0 as needed.

## Risks and mitigations

- **`swift-docc-plugin` becomes a transitive dependency for consumers.** It is a build-tool plugin, not a library product, so consumers do not link against it; verify by checking `swift package show-dependencies` after the Package.swift change.
- **Doc comment churn breaks unrelated PRs mid-rollout.** Mitigation: land the symbol pass in self-contained, file-scoped commits, not one giant PR. Reviewers can sign off subsystem-by-subsystem.
- **README inbound links break.** Mitigation: keep the README, slim its body but preserve all existing anchor IDs that are linked from external sources (release notes, blog posts).
- **`docs/` content drifts further from the public-facing story.** Accepted risk for 1.0; revisit if user-facing concepts and engineering notes start contradicting each other.
