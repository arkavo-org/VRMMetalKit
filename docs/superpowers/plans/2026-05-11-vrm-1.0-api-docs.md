# VRMMetalKit 1.0 API Documentation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a complete, navigable DocC-based API reference for VRMMetalKit alongside the 1.0 tag — Xcode Quick Help, a symbol graph, eight user-facing articles, and a hosted reference site at `https://arkavo-org.github.io/VRMMetalKit/`.

**Architecture:** Hybrid documentation strategy. Add a Swift-DocC catalog at `Sources/VRMMetalKit/VRMMetalKit.docc/` that owns the public-API reference and user-facing articles. Leave the existing `docs/` directory untouched as the engineering archive (ADRs, performance reports, internal specs). Wire `apple/swift-docc-plugin` for local preview and static-site generation, and publish via a new GitHub Actions workflow.

**Tech Stack:** Swift 6.2 / SPM, Swift-DocC, `apple/swift-docc-plugin` (1.4+), GitHub Actions, GitHub Pages.

**Spec:** [`docs/superpowers/specs/2026-05-11-vrm-1.0-api-docs-design.md`](../specs/2026-05-11-vrm-1.0-api-docs-design.md)

**Convention notes for the implementing engineer:**

- CLAUDE.md forbids "temporary contextual/informational comments in code." That rule targets noise like `// added for issue #123` or `// previously did X`. It does **not** apply to `///` doc comments — those are a load-bearing public-API contract and the entire point of this plan. Write them generously where the spec requires.
- Tests require `--disable-sandbox`. None of this plan adds tests, but if you re-run the suite to sanity-check, use the documented invocation in CLAUDE.md.
- Do **not** push without an explicit signal — see the user's memory note on Xcode Cloud CI credits. Commit freely throughout; the user will push when ready.
- Make commits small and self-contained (one subsystem per commit during the symbol pass) so reviewers can sign off section-by-section.
- **Topics-list parallel growth (applies to Tasks 3–8):** After documenting symbols in a subsystem, audit the corresponding `## Topics` group in `Sources/VRMMetalKit/VRMMetalKit.docc/VRMMetalKit.md`. Append any newly documented public symbol that is not already listed. Subdivide groups with `#### Subgroup` headings under `### Group` if a group exceeds ~15 entries. The single hard rule for Task 20 is **no orphan-symbol warnings**, and the cheapest way to honor it is incremental growth, not a Task-20 reconciliation. Include the landing-page edits in the same commit as the symbol-pass commit for that subsystem.

- [ ] **Step 1: Add the docc-plugin dependency**

Edit `Package.swift`. Add a `dependencies:` array (the package currently has none) and reference it from the library target's build plugins is **not** required — the docc plugin is a *package plugin* consumed via `swift package` commands, not a target plugin. We only need it in the top-level `dependencies` list.

Replace the existing `Package(...)` initializer's argument list so it includes a top-level `dependencies` array. The minimal edit:

```swift
let package = Package(
    name: "VRMMetalKit",
    platforms: [
        .macOS(.v26),
        .iOS(.v26)
    ],
    products: [
        .library(
            name: "VRMMetalKit",
            targets: ["VRMMetalKit"]
        )
    ],
    dependencies: [
        .package(url: "https://github.com/apple/swift-docc-plugin", from: "1.4.0")
    ],
    targets: [
        // ... existing targets unchanged
    ]
)
```

Leave all `targets` entries exactly as they are. The plugin attaches to `swift package` invocations, not to targets.

- [ ] **Step 1: Read every Core file in full**

Use the Read tool on each of the five files above. Do not skim — every `public` declaration needs a `///` summary, and you cannot write a useful summary without seeing the type's actual shape.

- [ ] **Step 2: Add doc comments using this exact comment shape**

For every `public` declaration (types, inits, methods, properties, cases), add a `///` summary directly above it. Use this style:

```swift
/// Loads a VRM model from a file URL using the default Metal device.
///
/// VRM 1.0 (`VRMC_vrm`) is preferred; VRM 0.0 files are converted on
/// the fly. Use ``VRMLoadingOptions`` to set quotas for untrusted input.
///
/// - Parameters:
///   - url: A file URL pointing to a `.vrm` or `.glb` file.
///   - device: The Metal device for GPU resource creation. If `nil`,
///     ``MTLCreateSystemDefaultDevice`` is used.
/// - Throws: ``VRMRendererError`` on invalid GLTF, unsupported VRM
///   version, or quota violations.
/// - Returns: A fully loaded, GPU-resident ``VRMModel``.
public static func load(from url: URL, device: MTLDevice? = nil) async throws -> VRMModel
```

Rules:
- One-line summary, period-terminated, leading verb ("Loads…", not "This loads…").
- Discussion paragraph **only** for types and entry points where behavior is non-obvious. For simple value-type properties, the one-line summary is enough.
- Document every `throws` case where the symbol throws.
- Use double-backtick references (``Symbol``) for cross-links to other VRMMetalKit types.
- Do **not** restate the type name in the summary ("The VRMModel struct represents…" is wrong; "A loaded VRM avatar with GPU-resident geometry and materials." is right).
- Do **not** add `// previously did X` or task-context comments — that violates CLAUDE.md.

For `VRMConstants` (the one public enum), document the enum itself and each `static let` constant. For the bone-name and expression-preset enums in `VRMTypes`, document the enum, the spec reason for the case set, and any cases whose names diverge from the wire format.

For `VRMLoadingOptions`, prioritize the deeper treatment — this is the security-sensitive surface. Include in the type's discussion:
- The default quotas and where they came from.
- Which limits matter for untrusted input (file size, texture count, vertex count, etc.).
- A short snippet showing how to construct a conservative options value.

For `StrictMode.swift` (`StrictLevel`, `RenderFilter`), document the three levels (`off`/`warn`/`fail`), when each is appropriate, and cross-link to <doc:StrictMode>.
